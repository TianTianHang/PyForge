## Context

PyForge 是一个基于 Tauri 的桌面 Python 开发环境，当前配置（PyPI 源、Python 安装源等）硬编码在 Rust 代码中。用户无法自定义这些配置，导致：
- 不同地区用户需要不同的镜像源
- 网络环境变化时无法灵活调整
- 企业用户可能需要使用内部源

本设计实现一个全局配置系统，支持用户自定义源配置和应用设置。

## Goals / Non-Goals

**Goals:**
- 支持 PyPI 镜像源配置
- 支持 Python 安装镜像源配置
- 支持 Python 下载策略配置
- 支持应用数据目录配置
- 支持 Jupyter 端口范围和超时时间配置
- 支持默认安装包列表配置
- 自动生成 uv.toml 配置文件
- 使用 `--config-file` 参数调用 uv 命令

**Non-Goals:**
- 前端 Settings UI（在下一个提案中实现）
- 数据目录迁移功能（在下一个提案中实现）
- 动态配置热更新
- 多用户配置隔离

## Decisions

### 1. 配置文件格式：TOML

**选择**: TOML
**理由**:
- Rust 生态中有成熟的 `toml` crate
- 语法简洁，易于人类编辑
- 支持嵌套结构，适合配置管理
- 与 uv 的配置格式一致

**替代方案**:
- JSON：不适合人类编辑
- YAML：缩进敏感，容易出错
- INI：不支持嵌套结构

### 2. 配置文件位置：`~/.pyforge/config.toml`

**选择**: 与数据目录一致
**理由**:
- 集中管理，所有 PyForge 相关文件在同一目录
- 便于迁移和备份
- 不污染系统配置目录

**替代方案**:
- `~/.config/pyforge/config.toml`：遵循 XDG 规范，但不利于数据迁移
- 环境变量指定：灵活性过高，不推荐

### 3. uv 配置文件：`~/.pyforge/uv.toml`，使用 `--config-file` 参数

**选择**: 在数据目录生成 uv.toml，使用 `--config-file` 参数
**理由**:
- 集中管理，与 PyForge 配置在同一目录
- 不影响系统其他 uv 使用
- 使用 `--config-file` 参数明确指定，避免歧义

**替代方案**:
- `~/.config/uv/uv.toml`：遵循 uv 用户级配置规范，但会影响系统 uv
- 环境变量：UV_CONFIG_FILE 环境变量，但需要在所有 uv 调用中设置

### 4. 路径管理改造：使用 static 全局变量

**选择**: 使用 `OnceCell` 存储配置的路径
**理由**:
- 线程安全
- 单例模式，全局共享
- 延迟初始化，支持启动时加载配置

**替代方案**:
- 每次调用时读取配置：性能开销大
- 传递配置参数：需要修改所有函数签名

### 5. 配置加载时机：应用启动时

**选择**: 在 `lib.rs` 的 `setup()` 中加载配置
**理由**:
- 一次性加载，后续使用全局变量
- 与现有启动流程一致
- 错误处理简单

**替代方案**:
- 按需加载：性能开销大
- 配置文件监听：复杂度高，非核心需求

### 6. 默认值策略：硬编码在 Default trait 实现中

**选择**: 在 `AppConfig` 的 `Default` trait 实现中定义默认值
**理由**:
- 代码清晰，易于维护
- 与 serde 配合良好
- 支持部分配置覆盖

**替代方案**:
- 配置文件模板：需要额外维护
- 环境变量覆盖：增加复杂度

## Risks / Trade-offs

### 风险 1: 配置文件损坏或格式错误

**风险**: 用户手动编辑配置文件可能导致格式错误
**缓解**: 
- 使用 serde 的严格解析，捕获错误
- 提供友好的错误提示
- 支持重置为默认配置

### 风险 2: uv 命令调用失败

**风险**: `--config-file` 参数可能导致 uv 调用失败
**缓解**:
- 保持向后兼容，config_path 为 Option
- 测试所有 uv 命令调用
- 提供降级方案（不使用配置文件）

### 风险 3: 配置文件路径问题

**风险**: 用户配置的数据目录不存在或不可写
**缓解**:
- 启动时检查并创建目录
- 验证目录可写性
- 回退到默认目录

### 权衡 1: 配置加载性能

**权衡**: 每次启动都读取配置文件
**影响**: 轻微性能开销
**接受**: 配置文件小，读取快，可接受

### 权衡 2: 配置文件与代码同步

**权衡**: 配置项变更需要更新代码和配置文件
**影响**: 版本升级时可能需要迁移配置
**接受**: 配置项稳定，变更频率低

## Migration Plan

### 部署步骤

1. 添加 `toml` 依赖到 `Cargo.toml`
2. 创建配置模型文件
3. 创建配置加载/保存逻辑
4. 创建 uv.toml 生成逻辑
5. 改造路径管理
6. 改造 uv 命令调用
7. 注册 Tauri 命令
8. 移除硬编码配置
9. 改造启动流程

### 回滚策略

- 保留硬编码常量作为默认值
- 配置文件不存在时使用默认值
- 配置解析失败时使用默认值

### 测试计划

1. 单元测试：配置模型序列化/反序列化
2. 集成测试：配置加载/保存
3. 端到端测试：uv 命令调用
4. 回归测试：现有功能不受影响

## Open Questions

1. **配置文件版本管理**：是否需要支持配置文件版本迁移？
   - 当前决策：不需要，配置项稳定，变更频率低

2. **配置文件备份**：是否需要自动备份配置文件？
   - 当前决策：不需要，用户可以手动备份

3. **配置验证**：需要验证哪些配置项？
   - URL 格式验证
   - 端口范围验证
   - 目录路径验证
