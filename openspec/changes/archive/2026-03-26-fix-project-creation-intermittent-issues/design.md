## Context

PyForge是一个桌面Python开发环境，使用Tauri 2.0构建。项目创建功能允许用户创建新的Jupyter项目并绑定Python环境。当前实现存在多个bug导致间歇性创建失败。

**当前状态**:
- 前端验证逻辑错误：检查环境名称而非项目名称
- 后端无并发保护：多个请求可能同时读写元数据文件
- 前端无防重复提交：用户可快速多次点击创建按钮
- 状态管理不一致：直接修改state对象

**约束**:
- 必须保持向后兼容
- 不能破坏现有的项目列表和删除功能
- 使用nix develop确保开发环境一致性

## Goals / Non-Goals

**Goals:**
1. 修复前端验证逻辑，正确检查项目名称是否存在
2. 添加前端防重复提交机制，改善用户体验
3. 修复后端竞态条件，确保元数据操作的原子性
4. 改进错误处理和用户反馈

**Non-Goals:**
1. 不改变项目创建的核心业务逻辑
2. 不引入新的数据库或存储系统
3. 不修改现有的项目模型或API接口
4. 不影响其他功能（环境管理、Jupyter启动等）

## Decisions

### 决策1：前端验证修复

**选择**: 修改`CreateProjectDialog`接收`projects` prop而非`environments`

**原因**:
- 当前代码错误地检查环境名称，导致逻辑错误
- 最简单的修复方案，只需修改prop传递

**替代方案**:
- 在`CreateProjectDialog`内部获取项目列表：增加复杂度
- 使用全局状态管理：过度设计

### 决策2：防重复提交机制

**选择**: 在`CreateProjectDialog`组件内部管理`isSubmitting`状态

**原因**:
- 局部状态管理，不影响其他组件
- React标准模式，易于理解和维护
- 自动在对话框关闭时重置状态

**替代方案**:
- 使用全局loading状态：增加复杂度
- 在父组件管理状态：需要额外prop传递

### 决策3：后端并发控制

**选择**: 使用文件锁（`fs2` crate）保护元数据操作

**原因**:
- 文件锁是跨平台的标准解决方案
- `fs2` crate成熟稳定，支持Windows/macOS/Linux
- 只在创建/删除项目时加锁，性能影响最小

**替代方案**:
- 使用内存锁（Mutex）：不跨进程，Tauri可能有多个窗口
- 使用SQLite：过度设计，增加依赖
- 原子文件操作：复杂度高，跨平台兼容性差

### 决策4：状态管理改进

**选择**: 使用不可变状态更新模式

**原因**:
- React最佳实践，避免状态更新不一致
- 减少潜在的bug
- 代码更清晰易维护

**替代方案**:
- 使用useReducer：过度设计
- 使用状态管理库：增加依赖

## Risks / Trade-offs

### 风险1：文件锁可能导致性能下降
**影响**: 创建项目时需要等待锁释放
**缓解**: 
- 只在关键路径加锁（元数据读写）
- 使用超时机制避免死锁
- 创建项目是低频操作，性能影响可接受

### 风险2：fs2依赖可能增加构建复杂度
**影响**: 需要在Cargo.toml中添加新依赖
**缓解**:
- fs2是轻量级依赖
- 已有大量项目使用，稳定性好
- nix develop环境确保依赖一致性

### 风险3：前端状态管理改变可能引入新bug
**影响**: 修改状态更新逻辑可能影响其他功能
**缓解**:
- 只修改CreateProjectDialog组件
- 保持现有API接口不变
- 充分测试

### 权衡1：文件锁 vs 数据库
选择文件锁而非数据库：
- 优点：简单、轻量、无额外依赖
- 缺点：不适合高并发场景（但项目创建是低频操作）

### 权衡2：局部状态 vs 全局状态
选择局部状态管理：
- 优点：简单、隔离性好
- 缺点：如果需要跨组件共享状态需要重构（但当前不需要）

## Migration Plan

### 部署步骤
1. 添加fs2依赖到Cargo.toml
2. 修改前端组件
3. 修改后端元数据操作
4. 运行测试确保无回归
5. 构建并测试完整应用

### 回滚策略
- 保持Git提交历史清晰
- 每个修复独立提交，可单独回滚
- 保留旧代码注释以便参考

### 验证方法
1. 单元测试：测试元数据操作的原子性
2. 集成测试：测试完整项目创建流程
3. 手动测试：快速点击创建按钮，验证防重复提交
4. 并发测试：模拟多个同时创建请求

## Open Questions

1. **是否需要添加fs2依赖？**
   - 需要检查现有依赖是否已有类似功能
   - 如果已有类似功能，优先使用现有依赖

2. **文件锁的超时时间应该设为多少？**
   - 建议：5秒（足够完成元数据操作）
   - 需要根据实际性能测试调整

3. **是否需要添加日志记录锁竞争情况？**
   - 开发阶段可以添加详细日志
   - 生产阶段可以减少或移除

4. **前端错误提示应该使用中文还是英文？**
   - 当前使用中文，保持一致性
   - 未来可考虑国际化支持
