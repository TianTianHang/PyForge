## Context

PyForge 当前采用 React + TypeScript + Tauri 架构，前端通过 `ProjectPanel` 组件管理项目列表和创建流程。现有的项目创建对话框 (`CreateProjectDialog`) 要求用户同时提供项目名称和环境选择，这对没有 Python 环境管理经验的学习者构成了认知障碍。项目列表 (`ProjectList`) 需要用户先选中项目，再点击"启动 Jupyter"按钮，形成了两步操作流程。

当前技术栈：
- 前端：React 19, TypeScript 5.8, Tailwind CSS
- 状态管理：React Hooks (useState, useEffect)
- 后端：Rust (Tauri 2.0)
- 构建：Vite 7, pnpm

**问题根源**：
1. 概念暴露过早："项目"和"环境"的关系对新手是额外认知负担
2. 缺乏默认值：首次使用需要做多项技术决策（Python 版本、包列表）
3. 操作步骤冗余：从打开应用到进入 Jupyter 至少需要 4 次点击
4. 无场景引导：基于技术配置而非使用场景进行选择

## Goals / Non-Goals

**Goals:**
- 将首次从打开应用到进入 Jupyter 的点击次数从 4+ 次减少到 2 次
- 为新手用户提供"开箱即用"的创建体验，无需理解环境概念
- 提供基于用途（数据分析、机器学习、Web 开发）的模板引导
- 保持向后兼容，高级用户可访问完整配置选项
- 建立用户经验分层机制（first-time / beginner / standard / advanced）

**Non-Goals:**
- 修改后端环境管理逻辑（Rust 代码不变）
- 改变项目或环境的数据模型
- 移除现有功能（所有当前配置选项保留）
- 引入新的外部依赖

## Decisions

### 1. 用户模式检测机制

**决策**：基于项目数量和历史行为自动检测用户经验水平

**理由**：
- 避免让用户手动选择"新手模式"或"高级模式"
- 0 个项目 = 首次用户，显示欢迎页和简化流程
- 1-3 个项目 = 初学者，保留简化选项但展示更多功能
- 4+ 个项目 = 标准用户，展示完整功能

**替代方案**：
- 手动模式切换：增加用户决策负担，未采用
- 仅基于首次启动标志：无法适应经验增长，未采用

### 2. 模板系统设计

**决策**：创建预配置的项目模板，每个模板包含：
- 模板名称和描述（面向用途）
- 默认 Python 版本（隐藏技术细节）
- 预装包列表（基于用途组合）
- 推荐使用场景说明

**初始模板**：
- `data-science`: 数据分析 (numpy, pandas, matplotlib, ipykernel)
- `machine-learning`: 机器学习 (numpy, pandas, scikit-learn, ipykernel)
- `web-development`: Web 开发 (flask, requests, ipykernel)
- `general-learning`: 通用学习 (ipykernel, jupyterlab)

**理由**：
- 基于用途而非技术配置，降低认知门槛
- 提供智能默认值，减少决策疲劳
- 模板可在项目设置中事后修改，容错性强

**数据结构**：
```typescript
interface ProjectTemplate {
  id: string;
  name: string;
  description: string;
  icon: string;
  pythonVersion: string;
  packages: string[];
  useCases: string[];
}
```

### 3. 创建对话框重构

**决策**：采用折叠式设计，默认显示简化视图

**简化视图（默认）**：
- 项目名称输入
- 模板选择（卡片式，可视化）
- "使用默认配置"提示
- 单一"创建并开始"按钮

**高级视图（折叠）**：
- 展开"自定义配置"后显示：
  - Python 版本选择
  - 环境选择（或创建新环境）
  - 自定义包列表
  - 所有原有选项

**理由**：
- 渐进式披露原则，新手不被复杂选项淹没
- 高级用户可快速访问完整控制
- 符合 UI/UX Pro Max 的"progressive-disclosure"最佳实践

### 4. 项目列表交互优化

**决策**：将"选中→启动"两步操作合并为"卡片直接启动"

**实现**：
- 移除选中状态（currentProjectId）的视觉区分
- 每个项目卡片右上角永久显示"启动 Jupyter"按钮
- 点击卡片主体→显示详情/设置
- 点击"启动"按钮→直接进入 Jupyter

**理由**：
- 减少操作步骤，从 4 步降到 2 步
- 符合用户心理模型（直接操作而非选中+操作）
- 保持高级功能入口（设置、删除）在卡片上

### 5. 空状态优化

**决策**：将空状态从"空白页+按钮"改为"欢迎引导"

**内容**：
- 友好的欢迎插图和标题
- 3 个快速开始模板卡片
- "了解更多"链接到文档（可选）
- "高级设置"折叠链接

**理由**：
- 提供明确的前进路径，而非让用户面对空白
- 通过模板卡片引导用户立即开始
- 降低"我不知道该做什么"的焦虑

### 6. 状态管理策略

**决策**：在 React 组件层管理用户模式，不引入全局状态管理库

**状态位置**：
- `userMode` 存储在 `ProjectPanel` 组件 state
- 通过 `projects.length` 和 localStorage 首次访问标志计算
- 模板配置存储为组件常量（未来可迁移到配置文件）

**理由**：
- 避免过度工程化，当前复杂度不需要 Redux/Zustand
- 状态仅用于 UI 展示，不涉及后端同步
- 保持简单，易于维护

**替代方案**：
- Tauri store：增加前后端同步复杂度，未采用
- URL 参数：桌面应用无需，未采用

### 7. 组件架构调整

**决策**：创建新组件，保持现有组件向后兼容

**新增组件**：
- `TemplateSelector.tsx`: 模板选择卡片组件
- `WelcomeGuide.tsx`: 首次启动欢迎页
- `QuickStartCard.tsx`: 快速开始卡片

**修改组件**：
- `CreateProjectDialog.tsx`: 添加模板选择和折叠逻辑
- `ProjectList.tsx`: 移除选中状态，添加直接启动按钮
- `ProjectPanel.tsx`: 添加用户模式检测逻辑

**理由**：
- 组件职责单一，易于测试和维护
- 现有组件通过 props 扩展，不破坏现有功能
- 新组件可独立开发和迭代

## Risks / Trade-offs

### Risk 1: 用户无法找到高级选项
**风险**：新手用户可能不知道展开"自定义配置"，导致无法访问高级功能

**缓解**：
- 在"创建并开始"按钮旁添加小字"高级设置"链接
- 项目设置中始终允许更换环境和包配置
- 添加工具提示或首次使用引导

### Risk 2: 模板不满足所有需求
**风险**：预定义模板可能无法覆盖所有使用场景

**缓解**：
- 提供通用学习模板作为兜底
- 高级视图允许完全自定义
- 未来可支持用户自定义模板（v0.4.0+）

### Risk 3: 用户模式判断不准确
**风险**：仅基于项目数量可能无法准确反映用户经验水平

**缓解**：
- 标准用户界面仍保留所有功能入口
- 用户可在设置中手动调整模式（未来版本）
- 持续监控用户行为，优化判断逻辑

### Trade-off: 简化 vs 灵活性
**权衡**：新手简化 vs 高级用户灵活性

**平衡**：
- 默认简化，高级功能折叠展示
- 所有原有功能完整保留
- 通过用户模式分层满足不同需求

### Trade-off: 开发时间 vs 用户体验
**权衡**：增加新组件和逻辑需要更多开发时间

**平衡**：
- 复用现有组件和样式
- 分阶段实施（MVP: 简化创建 → 优化列表 → 欢迎页）
- 设计文档确保实施方向清晰，减少返工
