## Context

当前系统使用符号链接来管理内核绑定：
- `kernel_links.rs` 负责在项目间创建符号链接
- `data_migration.rs` 处理迁移中的符号链接转换
- `config_commands.rs` 包含符号链接相关的复杂逻辑

这些代码涉及跨平台处理、权限检查、循环检测等复杂逻辑，导致维护困难。系统架构存在以下问题：
1. 跨平台代码分散且重复
2. Windows 平台需要特殊的权限检查
3. 符号链接的迁移逻辑复杂且容易出错
4. 缺少统一的文件操作抽象

## Goals / Non-Goals

**Goals:**
- 将内核链接从符号链接改为文件复制，简化代码逻辑
- 创建统一的文件操作抽象层，减少重复代码
- 移除所有平台特定的符号链接处理逻辑
- 降低系统权限要求，提高用户体验
- 保持现有功能的完整性（项目绑定、迁移等）

**Non-Goals:**
- 改变内核配置文件的结构或格式
- 修改 Jupyter 的内核发现机制
- 实现分布式文件系统支持
- 优化内核配置文件的大小或性能

## Decisions

### 1. 使用文件复制替代符号链接

**决策**: 将内核绑定从符号链接改为直接文件复制

**理由**:
- 内核配置文件 (`kernel.json`) 很小（~1-2KB），复制开销可忽略
- 消除符号链接的所有复杂性（权限、跨平台、循环检测）
- 提高数据隔离性，项目间完全独立
- 避免 Windows 平台的权限问题

**替代方案考虑**:
- 保持符号链接：需要继续维护复杂的跨平台逻辑
- 混合模式：增加系统复杂度，违背简化目标

### 2. 创建统一的文件操作抽象层

**决策**: 创建 `file_ops.rs` 模块，封装所有文件操作

**理由**:
- 减少重复代码（多个地方使用 `fs::copy`、`create_dir_all`）
- 提供一致的错误处理和日志记录
- 便于未来扩展和维护

**API 设计**:
```rust
pub struct FileOps;

impl FileOps {
    /// 复制目录及其内容
    pub fn copy_directory(src: &Path, dst: &Path) -> Result<(), FileError>

    /// 复制单个文件
    pub fn copy_file(src: &Path, dst: &Path) -> Result<(), FileError>

    /// 安全删除目录
    pub fn remove_directory(path: &Path) -> Result<(), FileError>
}
```

### 3. 迁移逻辑简化

**决策**: 移除迁移中的符号链接处理逻辑

**理由**:
- 新系统不依赖符号链接，无需特殊处理
- 简化迁移流程，提高可靠性
- 减少测试覆盖的复杂度

**实现细节**:
- 直接复制文件，无需路径转换
- 保持元数据的迁移逻辑不变
- 更新配置文件中的路径引用

### 4. 保持向后兼容性

**决策**: 通过迁移工具自动处理新旧系统转换

**理由**:
- 确保现有用户平滑升级
- 避免破坏性变更

**实现策略**:
- 迁移工具自动检测旧符号链接
- 自动转换为文件副本
- 更新相关配置

## Risks / Trade-offs

### 风险 1: 存储空间增加
**[风险]** → 每个项目存储内核配置副本，增加 ~1KB/项目
**缓解**: 1KB 对现代存储系统影响可忽略，且提高了可靠性

### 风险 2: 项目间数据隔离降低
**[风险]** → 原本通过符号链接共享，现在每个项目都有副本
**缓解**: 提高了数据隔离性，避免了符号链接损坏的问题

### 风险 3: 迁移过程中的数据丢失
**[风险]** → 迁移时可能破坏原有的符号链接结构
**缓解**: 迁移工具设计为"先复制后切换"模式，保留原始数据

### 权衡 1: 代码复杂度 vs 存储空间
**权衡**: 增加少量存储空间换取大幅降低的代码复杂度
**决策**: 接受，因为复杂性带来的维护成本远高于存储成本

### 权衡 2: 系统性能 vs 用户体验
**权衡**: 文件复制比符号链接稍慢，但用户感知差异小
**决策**: 优先考虑系统稳定性和用户权限要求

## Migration Plan

### 第一阶段：准备
1. 创建文件操作抽象层
2. 编写单元测试覆盖新的文件操作
3. 验证现有功能正常工作

### 第二阶段：内核绑定重构
1. 修改 `kernel_links.rs` 使用文件复制
2. 移除符号链接权限检查逻辑
3. 更新相关测试用例

### 第三阶段：迁移逻辑简化
1. 简化 `data_migration.rs`，移除符号链接处理
2. 更新 `config_commands.rs` 的 API 层
3. 实现迁移工具的向后兼容支持

### 第四阶段：测试和验证
1. 端到端测试所有迁移场景
2. 验证 Windows 平台兼容性
3. 性能回归测试

### 第五阶段：部署
1. 逐步部署到生产环境
2. 监控迁移日志和错误率
3. 收集用户反馈

## Open Questions

### Q1: 如何处理已经存在的符号链接？
**决议**: 迁移工具自动检测并转换为文件副本，保持透明转换

### Q2: 是否需要保留符号链接的兼容性选项？
**决议**: 不保留，完全转换为文件复制模式，简化长期维护

### Q3: 内核配置文件未来是否会变大？
**决议**: 即使配置变大，复制策略仍然有效。如需优化，可考虑未来的版本机制

### Q4: 如何验证迁移的正确性？
**决议**: 通过对比迁移前后的内核绑定状态，确保功能等效
